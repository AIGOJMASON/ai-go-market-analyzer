# STRUCTURAL FREEZE VS ACTIVATION POLICY

## Purpose

This policy defines the lawful difference between a frozen canon structure and live runtime activation work inside that structure.

It exists to prevent two errors:

1. treating lawful implementation work as if it were a canon violation
2. treating runtime experimentation as if it had authority to mutate canon boundaries

---

## Core Rule

After canon lock, **boundaries do not move without versioned canon change**.

Inside those frozen boundaries, **implementation may still be completed, hardened, or activated** as long as the work does not alter:

- layer membership
- authority ownership
- folder placement
- canonical responsibilities
- handoff direction
- registry scope
- command surface identity

---

## Official Activation Maturity States

### 1. `canon_frozen`
A surface or layer is `canon_frozen` when:

- its location is fixed
- its authority role is defined
- its boundary is documented
- its upstream and downstream connections are declared

This state means the structure is legitimate.
It does **not** mean the implementation is complete.
It does **not** mean the runtime path has been proven.

---

### 2. `implementation_active`
A surface or layer is `implementation_active` when:

- a real file exists at the canonical location
- the file contains live logic rather than placeholder text
- the file performs its declared bounded role
- the implementation stays inside frozen authority limits

This state means the surface is no longer a stub.
It does **not** mean the full transaction has been runtime-proven.

---

### 3. `runtime_proven`
A surface or route is `runtime_proven` when:

- it executes through the intended runtime path
- its governed artifacts are actually written or returned as required
- its post-step verification succeeds where applicable
- its completion state is reflected truthfully in continuity or receipts
- the route result matches the documented contract

This state means the route has passed a real execution proof.
It does **not** grant authority to expand into neighboring layers.

---

## Lawful Post-Lock Changes

The following are allowed after canon lock, as long as they remain inside the frozen boundary:

- replacing placeholders with live code
- tightening validation
- adding bounded helper functions
- improving failure handling
- adding receipts or diagnostics within owned scope
- documenting exact contracts for already-declared surfaces
- hardening persistence, verification, and continuity steps

---

## Illegal Post-Lock Changes Without Canon Revision

The following are not allowed without explicit canon revision:

- moving a module to another layer
- changing which layer owns an artifact
- introducing a new authority path that bypasses the declared handoff chain
- turning router into a hidden supervisory core
- turning monitoring into a mutation layer
- turning continuity into a command executor
- changing root structure, registry placement, or core hierarchy silently

---

## AI_GO Phase Interpretation Rule

For AI_GO runtime activation:

- canon tree lock established `canon_frozen`
- live module rewrites established `implementation_active`
- end to end research transaction execution establishes `runtime_proven`

These states must be tracked separately.
No document may use one of these terms as if it implies the others.

---

## Research Route Application

For the governed research route, the following are the maturity checkpoints:

1. `canon_frozen`
   - RESEARCH_CORE, runtime, monitoring, continuity, and packets layers are structurally locked

2. `implementation_active`
   - intake normalization exists
   - screening exists
   - trust assignment exists
   - packet persistence exists
   - watcher verification exists
   - SMI reflection exists

3. `runtime_proven`
   - command enters runtime
   - packet is persisted
   - watcher verifies persisted artifact
   - SMI records only after watcher success
   - runtime receipt truthfully reports final state

---

## Change Control Rule

If future work needs to change ownership, boundaries, or routing authority, that work is not an activation pass.
It is a canon change and must be versioned separately.

If future work only completes or hardens logic inside an already-frozen boundary, it remains activation work and does not mutate canon.

---

## Enforcement Summary

- Structure is frozen
- Activation is allowed inside frozen structure
- Runtime proof must be earned by execution
- No layer may absorb authority it was not given
- No completion claim may outrun execution truth
