# LOADER POLICY

## PURPOSE

Defines lawful loader behavior for AI_GO.

The loader establishes read order and system posture.

The loader does not execute.

The loader does not mutate.

The loader does not create authority.

## CHILD-CORE FLOW LAW

Any child-core reference loaded by the system must preserve:

source or provider signal  
→ RESEARCH_CORE  
→ engine curation  
→ adapter  
→ child-core ingress  

Provider retrieval is reserved for RESEARCH_CORE only.

Child-core context is bounded, curated, adapter-shaped, and advisory unless governed mutation is explicitly approved.

## LOADER MAY

- identify canon documents
- expose read order
- preserve authority boundaries
- preserve execution_allowed = false
- support operator understanding

## LOADER MAY NOT

- execute actions
- mutate workflow
- mutate project truth
- write memory
- write outcome feedback
- create child-core authority
- override governance

## MUTATION LAW

Persistence = Mutation.

If loader evidence, receipt, index snapshot, dashboard packet, memory record, outcome reference, or latest snapshot is persisted, it must be explicit, classified, traceable, and governed.

All mutation must pass:

State Authority  
→ Watcher  
→ Canon  
→ Request Governor  
→ Execution Gate  
→ Cross-Core Integrity  

## EXECUTION LAW

execution_allowed = false by default.

Loader activity cannot trigger execution.

## SUMMARY

The loader reads and exposes system posture only.

It cannot mutate, execute, or create authority.
