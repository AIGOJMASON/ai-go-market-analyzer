# PM TO LIB

## PURPOSE

Defines lawful PM interaction with LIB.

LIB is canon and documentation context.

LIB is not execution.

LIB is not mutation by default.

LIB does not create authority by being read.

## REQUIRED FLOW

Any PM-to-LIB path that references child-core context must preserve:

source or provider signal  
→ RESEARCH_CORE  
→ engine curation  
→ adapter  
→ PM interpretation  
→ LIB reference  
→ child-core ingress when routed  

Provider retrieval is reserved for RESEARCH_CORE only.

Child-core documentation context must remain curated, adapter-shaped, bounded, and advisory unless governed mutation is explicitly approved.

## PM MAY

- read LIB context
- reference canon policy
- preserve lineage metadata
- preserve authority boundaries
- preserve execution_allowed = false
- prepare lawful routing posture

## PM MAY NOT

- execute actions from LIB reference
- mutate LIB directly
- mutate workflow directly
- mutate project truth directly
- write memory outside governance
- create child-core authority
- override governance

## LIB MAY

- describe canon
- expose policy
- support operator understanding
- support validation

## LIB MAY NOT

- execute actions
- mutate state by being read
- authorize child-core action
- become runtime authority by itself

## MUTATION LAW

Persistence = Mutation.

If LIB evidence, PM evidence, receipt, memory record, outcome reference, dashboard packet, latest snapshot, index artifact, or documentation update is persisted, it must be explicit, classified, traceable, and governed.

All mutation must pass:

State Authority  
→ Watcher  
→ Canon  
→ Request Governor  
→ Execution Gate  
→ Cross-Core Integrity  

## EXECUTION LAW

execution_allowed = false by default.

PM-to-LIB reference cannot trigger execution.

## SUMMARY

PM-to-LIB is advisory canon reference only.

It cannot mutate, execute, retrieve providers, or create authority.
